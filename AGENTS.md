## 🤝 Протокол работы (Step-by-Step & Discussion-First)

Перед внесением любых изменений в код (кроме файлов планирования/описания):
1. **Анализ идеи**: Изучить шаг, предложенный пользователем.
2. **Оценка ресурсов**: Оценить риски потребления RAM/CPU, OOM (Out of Memory) и параллельной нагрузки.
3. **Обсуждение**: Предложить оптимизированное решение и обсудить его с пользователем.
4. **Интеграция**: Приступать к коду и тестам только после явного одобрения.
5. **Git и Коммиты**: Запрещено делать коммиты (как локальные, так и внешние `git push`) автоматически. Все внесенные изменения должны оставаться в рабочей копии (в незакоммиченном виде), чтобы пользователь мог наглядно видеть diff в своей IDE. Создание коммитов и отправка на GitHub выполняются строго по явному запросу пользователя.

*Исключение*: Если пользователь пишет слово **"подтверждаю"** (или **"confirm"**), агент может пропустить шаги 1–3 (анализ и обсуждение) и сразу перейти к реализации задачи на свое усмотрение.

## ⚡ Технические лимиты сервера (VPS ~1GB RAM, 1 vCPU)

* **Ограничение памяти**: Лимит PHP в Docker-контейнере жестко задан в `128M`.
* **Потоковая обработка**: Запрещено загружать большие выборки из БД или файлы в память целиком. Всегда использовать PHP-генераторы (`yield`), курсоры БД и постраничную обработку (chunks).
* **Скрапинг и сеть**: Все сетевые запросы (парсеры, API Telegram) должны иметь жесткие тайм-ауты (timeout <= 10s) во избежание зависания процессов.
* **Однопоточность**: Запрещено использовать многопоточность или запускать тяжелые сборки без ограничений памяти.

## 🗄️ Слой базы данных (PostgreSQL & Standalone Eloquent)

* **Подключение**: Настраивается лениво через `DatabaseServiceProvider` (`PDO::class` и `Illuminate\Database\Capsule\Manager` в DI-контейнере).
* **Жесткое правило Eloquent**: При выборках обязательно использовать `select()` для ограничения полей, а также методы `cursor()` (выборка по одной записи через генераторы) или `chunk()` вместо метода `get()`, чтобы удерживать расход памяти в пределах лимита `128M`.

## 🚀 Консольные миграции

Для изменения структуры БД используйте команду миграций:
```bash
docker exec mcp-telegram-editor php bin/console.php db:migrate
```
* **Местонахождение миграций**: `database/migrations/`.
* Каждый файл миграции должен возвращать анонимный класс с методами `up()` и `down()`, использующими `Illuminate\Database\Capsule\Manager as Capsule` для конструирования таблиц.

## 🛠️ Решение проблем индексации (LSP Intelephense / IDE)

* **Файл-помощник**: В корне проекта находится файл `_ide_helper.php` (добавлен в `.gitignore`), содержащий стабы (stubs) классов для корректного автодополнения и устранения ложных предупреждений типа `Undefined type`.
* **Правило дополнения (Исключительно по запросу)**: Дополнять `_ide_helper.php` новыми стабами классов/методов разрешено **исключительно по прямому запросу пользователя** в случае возникновения ошибок индексации. Самостоятельно и автоматически генерировать заглушки запрещено.
* **Общие проблемы индексации**: При появлении ложных ошибок `Undefined type` для установленных через Docker composer-зависимостей, которые физически есть на диске, алгоритм действий следующий (только по запросу пользователя):
  1. Добавить стаб класса/интерфейса в `_ide_helper.php` внутри соответствующего `namespace { ... }`.
  2. Предложить очистить кэш Intelephense (`rm -rf ~/.config/intelephense/workspace/*`) и перезапустить языковой сервер.
